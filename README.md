# Infra ナレッジベース

クラウドインフラ（AWS / Azure / Google Cloud）および生成AI活用（Claude）を、実務で使える観点で体系的にまとめるナレッジベース。

初学者向けの入門解説ではなく、設計・構築・運用で実際に判断が必要になるポイント（トレードオフ、落とし穴、運用上の注意点）を中心に記録する。

## 構成

| ディレクトリ | 内容 |
|---|---|
| [common/](common/README.md) | クラウド横断で共通する概念（ネットワーク、セキュリティ、IAM、監視、アーキテクチャパターン） |
| [aws/](aws/README.md) | AWS の主要サービス・設計/運用ポイント・資格 |
| [azure/](azure/README.md) | Azure の主要サービス・設計/運用ポイント・資格 |
| [google-cloud/](google-cloud/README.md) | Google Cloud の主要サービス・設計/運用ポイント・資格 |
| [ai/](ai/README.md) | Claude を中心とした生成AI活用（プロンプト設計、MCP、ガバナンス） |

## ドキュメントの書式ルール

各クラウドベンダーの README およびトピック別ドキュメントは、以下の順序で統一する。

1. **概要** — 何のためのサービス/領域か、全体像
2. **主要サービス** — 実務で頻出する主要サービス・機能の要点
3. **設計ポイント** — アーキテクチャ設計時に検討すべき事項、トレードオフ
4. **運用ポイント** — 構築後の運用・監視・コスト・トラブルシュートの観点
5. **ベストプラクティス** — 実務で得られた知見、アンチパターン、チェックリスト

資格関連のドキュメントは `<vendor>/certifications/` 配下にまとめ、試験情報・出題範囲・学習リソース・進捗管理の形式で記録する。

## 運用方針

- 一次情報（公式ドキュメント）で裏取りできない推測は書かない。不確実な内容は「要検証」と明記する。
- 汎用的な内容は `common/` に、ベンダー固有の内容は各ベンダーフォルダに書く。重複は避け、必要に応じて相互リンクする。
- 資格の学習メモは合格後も残し、更新履歴として活用する（試験は改定されるため取得日・バージョンを明記）。
