# Google Cloud IAM

## 概要

Google Cloudにおけるアクセス制御。組織 → フォルダ → プロジェクト → リソースの階層構造にIAMポリシーが継承される点が設計上の中心となる。

## 主要サービス

- Cloud IAM（ロール: 基本/事前定義/カスタムロール）
- リソース階層（Organization、Folder、Project）
- サービスアカウント（ワークロードの認証）
- Workload Identity連携（外部IdP/GKEワークロードとの連携）

## 設計ポイント

- 基本ロール（Owner/Editor/Viewer）は権限が広すぎるため、事前定義ロールまたはカスタムロールを使用する。
- 権限はプロジェクト単位よりも、可能な限り細かいリソース単位（フォルダ、個別リソース）で付与し影響範囲を絞る。
- サービスアカウントキーの発行は避け、Workload IdentityやImpersonationを優先する。

## 運用ポイント

- IAM Recommenderで過剰権限を検出し、定期的に是正する。
- サービスアカウントの棚卸し（未使用アカウントの無効化）を運用に組み込む。
- 組織ポリシー（Organization Policy Constraints）でサービスアカウントキー作成の禁止等を強制する。
