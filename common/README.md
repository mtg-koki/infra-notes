# common — クラウド横断の共通知識

## 概要

AWS / Azure / Google Cloud に共通する概念・設計思想をまとめる領域。ベンダー固有の実装詳細は各ベンダーフォルダに書き、ここでは「クラウドが変わっても通用する考え方」に絞る。

## 主要トピック

| ファイル | 内容 |
|---|---|
| [networking.md](networking.md) | VPC/VNet設計、セグメンテーション、接続方式（VPN/専用線/ピアリング）の考え方 |
| [security.md](security.md) | 境界防御からゼロトラストまで、暗号化・脆弱性管理・インシデント対応の共通原則 |
| [iam.md](iam.md) | 認証・認可の共通モデル、最小権限、権限設計のアンチパターン |
| [monitoring.md](monitoring.md) | ログ・メトリクス・トレースの設計、SLI/SLO、アラート設計 |
| [architecture-patterns.md](architecture-patterns.md) | マルチクラウド/ハイブリッド構成、DR、代表的なアーキテクチャパターン |

## 設計ポイント

- ベンダー固有の機能に依存しすぎると移植性・交渉力が下がる。抽象化すべき層と、ベンダー機能を積極活用すべき層を切り分ける。
- 「クラウド横断で同じ設計思想を適用できるか」を判断基準にする（例: IAMの最小権限原則は全ベンダー共通）。

## 運用ポイント

- 複数クラウドを併用する場合、監視・ログ基盤は極力共通化し、運用負荷を下げる。
- ベンダーごとに用語が異なる点（例: VPC vs VNet、IAMロール vs マネージドID）は対応表を整備する。

## ベストプラクティス

- ここに書く内容は「2つ以上のベンダーで再利用できる知見」に限定し、陳腐化を防ぐ。
- 各ベンダーフォルダのドキュメントからは、対応する共通トピックへ相互リンクする。
