# Infra リポジトリ運用ルール

## このリポジトリについて

AWS/Azure/Google Cloud/AIに関する個人の学習・実務メモを管理するリポジトリ。共同編集者はおらず、CI/CDも組んでいない個人用リポジトリ。

## Git運用（自動実行してよい範囲）

- 同じファイルを試行錯誤しながら編集している間（ユーザーからの修正依頼が続いている間）は、都度コミットしない。ユーザーがその内容に一旦納得した区切りでまとめて `git add <対象ファイル>` → `git commit` する。
- コミット後は都度 `git push origin main` してよい。個人用リポジトリで他者への影響がないため、プッシュ前の確認は不要。ただしコミット自体は上記の通り編集が落ち着いてから行う。
- ただし以下は事前確認する。
  - `git add -A` / `git add .` など対象を絞らない一括ステージ（意図しないファイル混入を避けるため、変更したファイルを明示的に指定する）
  - force push、履歴改変（rebase -i、reset --hardなど）
  - `.gitignore` に該当しそうな秘匿情報を含むファイルの追加
- コミットメッセージは変更内容が分かる一言で簡潔に書く。
- 詳細なコマンド手順は [common/git-workflow.md](common/git-workflow.md) を参照。

## ドキュメント編集の方針

- 既存ファイルの構成・見出しスタイル（概要/主要トピック/設計ポイント/運用ポイント/ベストプラクティス等）に合わせる。
- 新しいトピックを追加した場合は、該当フォルダの `README.md` の一覧表にもリンクを追加する。
- 初稿から簡潔さを優先する。長い説明文より、表または一行の箇条書きを優先する（冗長になりがちなので、書いた後に「削れる説明はないか」を自己チェックする）。
- 個別サービスのドキュメント（例: aks.md, agw.md）は「概要 → 基本用語 → 設定/パラメーター → Tips・補足 → ベストプラクティス」の順で構成すると読みやすい。

## 添付ファイルの扱い

- ユーザーが添付したテキストファイルの内容が文字化けしている、または空に見える場合、貼り付けテキストをそのまま使わず、実ファイルパスを教えてもらいReadツールで直接読み込む。貼り付け経由だと制御文字が失われ、機械的な文字化け修正（エンコーディング変換）ができないことがある。
